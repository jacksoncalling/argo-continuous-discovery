# Solution Ideation — Teresa Torres' Framework

## The Rule: Compare, Don't Evaluate

Torres requires comparing at least 3 solutions for a target opportunity. This is grounded
in decision-making research: evaluating solutions one at a time produces worse decisions
than comparing multiple options side by side.

**When to generate more than 3:**
- The opportunity is high-risk (wrong solution is costly)
- The opportunity is a differentiator (commodity solutions won't win)
- The team wants to innovate (novel approaches need a wider search)

## What Counts as a Solution

"A solution can be a product, a feature, a service, a workflow, a process, documentation,
or anything else" that addresses an opportunity (Torres).

This is deliberately broad. A solution to "customers can't remember product recommendations
after leaving the spa" could be:
- A feature (digital recommendation card sent via SMS)
- A process change (spa prints a take-home card)
- A service (follow-up email with recommendations 24h later)
- Documentation (product guide QR code on the shelf)

Don't constrain to software features. The best solution might not be code.

## Solution Quality Criteria

Each solution should be:

| Criterion | Test |
|-----------|------|
| **Distinct** | Would a customer experience this differently from the other solutions? |
| **Concrete** | Can you describe what the customer sees/does/gets? |
| **Traceable** | Can you point to transcript evidence that this would address the need? |
| **Assumption-rich** | Can you identify what must be true for this to work? |

## Identifying the Key Assumption

Every solution rests on assumptions. The key assumption is the one that, if wrong,
invalidates the entire solution. This is what gets tested in `05-experiments/`.

Torres categorizes assumptions as:

| Category | Question |
|----------|---------|
| **Desirability** | Do customers want this? |
| **Usability** | Can customers figure out how to use this? |
| **Feasibility** | Can we build this? |
| **Viability** | Does this work for the business? |
| **Ethical** | Should we build this? |

The riskiest assumption is the one to test first.
